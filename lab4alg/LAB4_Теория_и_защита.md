## 1. Базовые понятия: граф

**Граф** \(G = (V, E)\):  
- \(V\) — множество **вершин** (в программе нумеруются **0, 1, …, n−1**, где `n = |V|`).  
- \(E\) — множество **рёбер**.

**Неориентированное ребро** соединяет две вершины «без стрелки»: можно ходить в обе стороны.  
**Ориентированное ребро** (дуга) идёт **из** вершины **в** вершину — по нему разрешён переход только в направлении стрелки.

**Простой граф** в нашей реализации: нет **петель** (ребра из вершины в саму себя отбрасываются) и нет **кратных** рёбер (одна и та же пара учитывается один раз).

**Путь** от \(A\) до \(B\) — последовательность вершин, где соседние соединены ребром (с учётом ориентации в орграфе).  
**Длина пути в невзвешенном графе** — число **рёбер** на пути (не число вершин).

**Кратчайший путь** в невзвешенном графе — путь с **минимальным числом рёбер**.

## 2. Представления графа в памяти (теория + код)

Задание требует уметь **вывести** граф в четырёх формах. Все они эквивалентны по информации, но по-разному удобны для алгоритмов и для человека.

### 2.1. Матрица смежности

**Идея:** таблица `n × n`. Элемент `(i, j) = 1`, если есть ребро **из i в j**, иначе `0`.

- Неориентированный: матрица **симметрична** (если есть `i—j`, есть и `j—i` в смысле смежности).
- Ориентированный: в общем случае **несимметрична**.

**В коде:** класс `Graph`, метод `adjacencyMatrix()`.  
Строится проходом по списку рёбер `getEdges()`: для каждого ребра выставляются единицы; для неориентированного — зеркально `i,j` и `j,i`.

**Плюсы:** проверка «есть ли ребро i→j» за O(1).  
**Минусы:** память **O(n²)** даже если рёбер мало.

---

### 2.2. Матрица инцидентности

**Идея:** строки — **вершины**, столбцы — **рёбра** (каждому столбцу соответствует одно ребро в **фиксированном порядке** — у нас порядок как в `getEdges()`).

- **Неориентированный граф:** в столбце ребра `{u,v}` стоят **1** в строках `u` и `v`, остальные **0** (иногда в курсах ставят 2 — это другая конвенция; важно **явно сказать**, какую вы выбрали).
- **Ориентированный граф (частый вариант):** в столбце дуги `u → v` в строке **u** стоит **+1** (исход), в строке **v** — **−1** (заход).

**В коде:** `Graph.incidenceMatrix()`.

**Плюсы:** удобно в теории графов и в некоторых задачах линейной алгебры.  
**Минусы:** размер **O(n · m)**, где `m = |E|`; для плотных графов велико.

---

### 2.3. Список смежности

**Идея:** для каждой вершины `i` хранится список вершин `j`, в которые можно попасть **одним ребром из i** (для неориентированного — соседи).

**В коде:** внутри `Graph` поле `adjacency`; наружу:
- `adjacencyList()` — **копия** (безопасно менять снаружи);
- `adjacencyView()` — **неизменяемый вид** без копирования соседей (удобно для BFS/DFS — быстрее).

**Плюсы:** память **O(n + m)**; обход соседей вершины за O(степень вершины).  
**Минусы:** проверка ребра `i,j` может потребовать перебора списка.

---

### 2.4. Список рёбер

**Идея:** просто перечень пар `(from, to)`.

**В коде:** `Graph.edgeList()` и тип `Edge` в `Edge.java` (`record`: неизменяемая пара индексов).

**Плюсы:** компактно для ввода/вывода и для генератора.  
**Минусы:** поиск соседей не такой быстрый, как у списка смежности, если не строить дополнительные структуры.

---

### 2.5. Разделение «модель» и «генератор»

**Требование задания:** класс графа **не** занимается случайной генерацией, только **хранит** данные и отдаёт представления.

- **Модель:** `Graph` + `Edge`.
- **Генерация:** `RandomGraphGenerator` строит список рёбер и передаёт его в конструктор `Graph`.

## 3. Генератор случайного графа (теория + код)

**Цель:** получить **простой** граф со случайной топологией, но с **ограничениями**, заданными в лабораторной.

### 3.1. Параметры (как в `GeneratorConfig`)

| Параметр | Смысл |
|----------|--------|
| `minVertices` … `maxVertices` | Случайно выбирается `|V|` в этом отрезке (целые). |
| `minEdges` … `maxEdges` | Случайно выбирается **целевое** число рёбер `eTarget`. |
| `maxVertexDegree` | Верхняя граница степени (см. ниже для орграфа). |
| `directed` | `false` — неориентированный, `true` — ориентированный. |
| `maxInDegree`, `maxOutDegree` | Для орграфа: сколько рёбер может **входить** / **выходить** из вершины. |

Фабрики `undirected(...)` и `directed(...)` упрощают создание конфигурации.

### 3.2. Алгоритм генерации (`RandomGraphGenerator.generate()`)

1. Случайно выбрать `v` и `eTarget` по диапазонам.  
2. Пока набрано меньше `eTarget` рёбер и не исчерпан лимит попыток:
   - выбрать случайную пару вершин `(a, b)`, `a ≠ b`;
   - если ребро уже есть — пропуск;
   - проверить **ограничения степеней**;
   - добавить ребро и обновить счётчики степеней.

**Неориентированный случай:** ребро хранится как неупорядоченная пара `(min(a,b), max(a,b))`, ключ в `Set` — строка `"u-w"`, чтобы не дублировать.

**Ориентированный случай:** пара упорядочена `(a → b)`, ключ `"a->b"`. Проверяются:
- `outDeg[a] < maxOutDegree`, `inDeg[b] < maxInDegree`;
- суммарная степень (вход+исход) у концов не превышает `maxVertexDegree` (дополнительное ограничение «общей нагрузки» на вершину).

**Важно для защиты:** если ограничения слишком жёсткие или `eTarget` завышен, цикл может **закончиться раньше** — фактическое `|E|` будет **меньше** цели. Это оговорено в комментарии к классу.

## 4. Поиск кратчайшего пути: BFS (теория + код)

### 4.1. Идея обхода в ширину (Breadth-First Search)

Стартуем из `start`. Сначала обрабатываем все вершины на **расстоянии 1** ребро, потом на **2**, и т.д. Используется **очередь** FIFO: «сначала пришли — сначала обработали».

**Свойство для невзвешенного графа:** когда вершина `end` **впервые** достигнута, число пройденных рёбер от `start` **минимально** среди всех путей. Поэтому BFS находит **кратчайший путь по числу рёбер**.

### 4.2. Реализация (`BfsShortestPath.find`)

- Очередь `ArrayDeque`, массив `seen[]` — уже посещённые.
- Массив `parent[v]` — **откуда** мы впервые попали в `v` (для восстановления пути). Неинициализированные значения: `-1`.
- После обхода: если `end` не помечен — пути нет (`PathResult.none()`).
- Иначе идём от `end` по `parent` к `start`, разворачиваем список — получаем путь от `start` к `end`.

**Сложность:** **O(|V| + |E|)** — каждая вершина и каждое ребро (в списке смежности) рассматриваются ограниченное число раз.

**Особый случай:** `start == end` — путь длины 0, возвращается одна вершина.

## 5. DFS и кратчайший путь: почему в коде IDDFS (`DfsShortestPath`)

### 5.1. Наивный одиночный DFS

Классический DFS «иди вглубь, пока не упрёшься» при **первом** попадании в `end` **не гарантирует** кратчайший путь: можно сначала уйти «в длинный обход», хотя короткий путь существует.

### 5.2. Итеративное углубление (IDDFS)

**Идея:** выполнять серию **ограниченных по глубине** DFS:

- ограничение 1: есть ли путь из `start` в `end` не длиннее **1** ребра?
- ограничение 2: не длиннее **2** рёбер?
- … пока не найдём путь или не исчерпаем разумный максимум.

Первое сработавшее ограничение даёт путь с **минимальным** числом рёбер — то же, что BFS в невзвешенном графе.

В коде `limitedDfs` — это DFS с параметром «сколько рёбер ещё можно пройти» (`depthLeft`). Массив `onPath` не даёт зациклиться на уже стоящих в текущем пути вершинах (**простой путь**).

**Верхняя граница лимита:** в простом пути не больше **|V|−1** ребра — цикл `for (limit = 1 … n−1)`.

**Сложность:** в худшем случае **хуже**, чем у BFS (много повторных обходов), на практике на плотных графах BFS обычно **быстрее** — это хороший аргумент в **анализе** лабораторной.

---

## 6. Результат поиска пути (`PathResult`)

`Optional<List<Integer>>` — либо путь (список вершин по порядку), либо пусто.  
Длина в рёбрах: для пути из `k` вершин — **`k−1` ребро**.  
Если пути нет — `Optional.empty()` и длина **−1** (условное соглашение в коде).

Фабрики `none()` и `of(...)` централизуют создание результата.

## 7. Экспериментальная часть (теория + код в `Main`)

### 7.1. Что требует задание

- **10 графов** с **растущими** `|V|` и `|E|` (масштаб под возможности ПК).
- Для каждого — **случайные** вершины `A` и `B`.
- Найти **кратчайший путь** или установить отсутствие пути.
- Сравнить **BFS** и **DFS** (у нас DFS в форме **IDDFS**) по **времени**.
- Построить **график** и дать **анализ**.

### 7.2. Как сделано в `Main`

- `runTenGraphBenchmark()`: цикл `i = 0…9`, для каждого `i` задаются диапазоны `vMin…vMax`, `eLo…eHi`, `maxDeg`, вызывается генератор.
- `Random(20250322L)` — **фиксированное зерно** для **воспроизводимости** отчёта (можно объяснить на защите).
- Вершины `a`, `b` выбираются случайно, **`b ≠ a`**.
- Время: `System.nanoTime()` до и после вызова алгоритма.
- Проверка: длины путей BFS и IDDFS **совпадают** (иначе исключение — защита от ошибки реализации).
- `warmUp()` — прогрев JVM (чтобы первые замеры не были «холодными»; на защите можно сказать, что это снижает шум).

### 7.3. График (`TimingChart`)

Библиотека **JFreeChart**: столбчатая диаграмма, по оси X — подпись графа (`|V|`, `|E|`), по Y — время в **микросекундах** (наносекунды / 1000) для наглядности.

### 7.4. Запись строки эксперимента (`BenchmarkRow`)

Поля: подпись, `|V|`, `|E|`, `from`, `to`, время BFS, время DFS, длины путей (для контроля).

## 9. Типичные вопросы на защите (короткие ответы)

**Почему BFS даёт кратчайший путь?**  
Потому что вершины достигаются по **слоям** расстояния от старта; первое попадание в `end` — на минимальном числе рёбер.

**Почему не просто один DFS?**  
Один DFS до первого найденного `end` **не обязан** быть кратчайшим. IDDFS исправляет это, перебирая максимальную длину пути.

**Память матрицы смежности?**  
O(n²). Список смежности — O(n + m).

**Что если пути нет?**  
BFS не достигнет `end`; IDDFS исчерпает все лимиты — оба возвращают «нет пути».

**Почему на графике DFS часто дольше?**  
IDDFS много раз **повторно** обходит фрагменты графа при увеличении лимита глубины.
