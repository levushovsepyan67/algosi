## Теория к лабораторной работе №3 (Стек)

Ниже приведена теоретическая часть, которая помогает понимать, что реализуется в лабораторной работе: что такое шаблоны/обобщения (generics), что такое стек, чем стек отличается от очереди, какие бывают реализации, сложности операций и типичные применения.

---

### 1. Контейнеры и абстрактные типы данных (ADT)

**Контейнер** — структура данных, предназначенная для хранения и управления набором элементов. Важная идея: контейнер определяет не только хранение, но и допустимые операции (добавление, удаление, просмотр, обход и т.д.).

**Абстрактный тип данных (ADT)** — описание структуры данных через операции и их поведение, без привязки к конкретной реализации. Например, «стек» как ADT описывает операции `push/pop/peek`, но не говорит, будет ли это массив или список.

---

### 2. Обобщения (шаблоны, generics) в Java

В Java для «шаблонного» (типо‑независимого) кода используется механизм **обобщений (generics)**. Он позволяет писать контейнер, который работает с **произвольным типом**, например:

- `SimpleStack<Integer>` — стек целых чисел,
- `SimpleStack<String>` — стек строк,
- `SimpleStack<Person>` — стек объектов.

**Зачем это нужно:**

- чтобы один и тот же код контейнера работал для разных типов данных;
- чтобы компилятор проверял типы и предотвращал ошибки (не нужно вручную приводить типы);
- чтобы обеспечить удобство и читаемость кода.

**Отличие от «сырых» типов и Object:**

- если хранить всё как `Object`, придётся делать приведения типов, что опасно;
- generics позволяют компилятору гарантировать корректность типов.

Ограничение Java: generics реализованы через **type erasure** (стирание типов), то есть во время выполнения JVM обычно не хранит информацию о параметре типа `T` так же, как в compile‑time.

---

### 3. Стек: определение и принцип работы

**Стек (stack)** — структура данных, работающая по принципу **LIFO** (*Last In — First Out*): последний добавленный элемент извлекается первым.

Основные операции:

- `push(x)` — положить элемент `x` на вершину стека;
- `pop()` — снять (удалить и вернуть) элемент с вершины;
- `peek()` — вернуть элемент с вершины без удаления;
- `isEmpty()` — проверка на пустоту;
- `size()` — количество элементов.

Стек можно представить как стопку тарелок: добавляем сверху — и снимаем тоже сверху.

---

### 4. Очередь и отличие стека от очереди

**Очередь (queue)** — структура данных с принципом **FIFO** (*First In — First Out*): первый добавленный элемент извлекается первым.

Основные операции очереди:

- `enqueue(x)` — добавить элемент в конец очереди;
- `dequeue()` — извлечь элемент из начала;
- `peek()` — посмотреть элемент в начале;
- `isEmpty()`, `size()`.

**Главное отличие:**

- стек: LIFO — «последний вошёл, первый вышел»;
- очередь: FIFO — «первый вошёл, первый вышел».

**Пример:**

- стек полезен для отката (undo), рекурсии, парсинга скобок;
- очередь полезна для обработки задач в порядке поступления (планировщики, очереди сообщений).

---

### 5. Реализации стека

Стек как ADT можно реализовать разными структурами:

#### 5.1. Стек на массиве

Идея: хранить элементы в массиве, а вершину задавать индексом `size - 1`.

Плюсы:

- обычно быстрее из‑за хорошей **локальности памяти** (элементы лежат подряд);
- простая реализация, минимум накладных расходов.

Минусы:

- нужен механизм расширения массива, если места не хватает (динамический массив);
- при расширении требуется копирование в новый массив, поэтому `push` имеет **амортизированную** сложность O(1) (обычно редко бывает O(n) при расширении).

#### 5.2. Стек на односвязном списке

Идея: каждый элемент — это узел `Node(value, next)`, где `head` указывает на вершину стека.

Плюсы:

- не нужно заранее знать размер, расширение происходит по одному элементу;
- `push/pop` всегда O(1) без копирования массива.

Минусы:

- каждый элемент хранится в отдельном узле, это дополнительные ссылки и объекты;
- хуже локальность памяти → часто немного медленнее массива;
- больше накладные расходы по памяти на один элемент.

---

### 6. Сложность операций

Для обеих реализаций основные операции имеют следующие оценки:

- `push`:
  - массив: O(1) амортизированно (иногда O(n) при расширении);
  - список: O(1).
- `pop`: O(1).
- `peek`: O(1).
- `isEmpty`, `size`: O(1).

**Амортизированная сложность** означает, что дорогие операции (например, расширение массива) случаются редко, поэтому средняя стоимость операции по большой серии вызовов остаётся O(1).

---

### 7. Итератор и обход элементов

**Итератор** — объект, который позволяет последовательно получать элементы контейнера без раскрытия внутренней структуры.

В Java стандартный подход — реализовать интерфейс `Iterable<T>`, тогда контейнер можно обходить так:

```java
for (T x : container) {
    // обработка x
}
```

Для стека чаще всего удобно обходить элементы **от вершины к основанию**, потому что именно вершина является «активным» концом структуры.

---

### 8. Типичные применения стека

Стек используется во многих задачах:

- обработка рекурсии и вызовов функций (call stack);
- отмена действий (undo/redo);
- проверка правильности скобочных последовательностей;
- вычисление выражений (постфиксная/инфиксная запись, алгоритм Дейкстры «shunting-yard»);
- поиск в глубину (DFS) в графах (итеративная версия);
- разворот последовательности (перекладывание элементов).

---

### 9. Память и практические различия реализаций

При сравнении реализаций важно учитывать:

- **Скорость**:
  - массив обычно быстрее из‑за кэш‑дружелюбности;
  - список может быть медленнее из‑за разрозненных узлов в памяти.
- **Память**:
  - массив хранит элементы компактно;
  - список требует памяти на узлы (ссылки `next`) + накладные расходы объектов.
- **Рост структуры**:
  - массив расширяется «скачками» (обычно ×2), поэтому иногда выделяет больше памяти, чем нужно прямо сейчас;
  - список растёт равномерно по одному элементу.

Выбор реализации зависит от требований: если важна максимальная скорость и известны (или ограничены) размеры — удобен массив; если важна гибкость и гарантированное O(1) на вставку без расширения — удобен список.
