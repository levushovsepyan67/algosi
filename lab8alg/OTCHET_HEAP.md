 # Отчёт: лабораторная работа по кучам (бинарная и биномиальная min-куча)

## 1. Описание метода и модели

Ниже приведены **определения и теоретическая база** структур, которые реализованы в работе. Формулировки можно использовать как справочный материал к коду.

---

### 1.1. Дерево (общее определение)

**Дерево** — это связный неориентированный граф без циклов, в котором выделена одна вершина **корень**. Остальные вершины организованы **уровнями**: у каждой вершины (кроме корня) ровно одна вершина **родитель** на предыдущем уровне; вершины, соединённые с данной ребром «вниз от корня», называются **детьми**. **Высота** вершины — длина самого длинного пути от неё к листу; **высота дерева** — высота корня. **Глубина** вершины — число рёбер на пути от корня к этой вершине.

---

### 1.2. Бинарное дерево

**Бинарное дерево** — дерево, у каждой вершины которого не более **двух** детей; их называют **левым** и **правым** потомком (если ребёнок один, второй считается отсутствующим).

**Полное бинарное дерево** — дерево, в котором на всех уровнях, кроме последнего, вершин «максимум», а на последнем уровне листья заполнены **слева направо без пропусков**. Такое дерево удобно хранить **в массиве** без явных указателей: если вершины пронумерованы сверху вниз, слева направо, начиная с 0, то для индекса `i` родитель — `(i − 1) / 2` (целочисленно), левый ребёнок — `2i + 1`, правый — `2i + 2`.

**Бинарное дерево поиска (BST)** — бинарное дерево, в котором у каждой вершины хранится **ключ**, и все ключи **левого** поддерева меньше ключа вершины, а **правого** — больше. (В данной лабораторной BST не реализуется, но определение полезно для контекста: **куча** — другая структура на бинарном дереве, задаётся не порядком обхода BST, а **свойством кучи**.)

---

### 1.3. Куча (heap) и бинарная куча

**Куча (heap)** — это способ организации данных в виде дерева с дополнительным **инвариантом приоритета** между родителем и детьми.

**Бинарная куча** — куча, реализованная как **полное бинарное дерево** (часто — в массиве, как в п. 1.2).

**Min-куча (min-heap)** — бинарная куча, в которой выполняется **свойство min-кучи**: ключ в каждой вершине **не больше** ключей в обоих детях (если они есть). Следствие: **глобальный минимум** всегда находится в **корне** дерева; в массиве это элемент с индексом `0`.

**Основные операции (типичная асимптотика по числу элементов `N`):**

- просмотр минимума — **O(1)** (чтение корня);
- вставка — **O(log N)** (добавление в конец и «просеивание вверх»);
- удаление минимума — **O(log N)** (замена корня и «просеивание вниз»).

**Память:** **O(N)** под массив ключей; накладные расходы малы, доступ к соседям по индексу — **хорошая локальность** в памяти.

---

### 1.4. Биномиальное дерево

**Биномиальное дерево порядка `k`** (обозначают `Bₖ`) определяется **рекурсивно**:

- `B₀` — одна вершина;
- `Bₖ` (`k ≥ 1`) получается из дерева `Bₖ₋₁` добавлением **ещё одного** дерева `Bₖ₋₁` как поддерева: корень одного `Bₖ₋₁` становится **самым старшим ребёнком** корня другого.

**Свойства:** в `Bₖ` ровно **`2ᵏ`** вершин; высота дерева равна **`k`**; у корня степень (число детей) равна **`k`**, дети по убыванию размеров — корни деревьев `Bₖ₋₁`, `Bₖ₋₂`, …, `B₀`.

Для **min-**варианта на рёбрах «родитель–ребёнок» дополнительно требуют: ключ родителя **не больше** ключей в поддеревьях (как в куче).

---

### 1.5. Биномиальная куча (binomial heap)

**Биномиальная куча** — это набор (мультимножество) биномиальных деревьев `B₀`, `B₁`, …, удовлетворяющий условию: **не более одного** дерева каждого порядка `k` одновременно (как разряды в **двоичной записи** числа `N`: каждый бит — «есть или нет» дерево `Bₖ`).

Корни деревьев хранят в **списке** (часто — по возрастанию порядка). **Слияние** двух биномиальных куч аналогично **поразрядному сложению** двоичных чисел: при совпадении порядков два дерева **склеиваются** в одно дерево следующего порядка (с сохранением min-свойства выбором корня с меньшим ключом).

**Операции (типичные оценки):**

- слияние двух куч — **O(log N)** по числу различных порядков корней;
- вставка одного элемента (как куча из `B₀`) и слияние — **амортизированно O(1)**, в худшем за одну операцию — **O(log N)**;
- поиск минимума — **O(log N)** (нужно сравнить корни всех деревьев в списке; деревьев **O(log N)**);
- удаление минимума — **O(log N)** (найти корень с минимумом, удалить, объединить оставшиеся корни с «лесом» из детей удалённого корня).

**Память:** **O(N)** узлов с указателями; накладные расходы выше, чем у массива в бинарной куче; зато структура **естественно поддерживает быстрое слияние** куч (mergeable heap).

---

### 1.6. Модель эксперимента (связь с лабораторной)

Сравниваются **две реализации min-кучи** на одних и тех же наборах ключей. Для каждого `N = 10^i`, `i = 3…7`, после заполнения кучи **1000** раз измеряются операции `findMin`, затем на **новой** куче того же размера — **1000** `deleteMin`, затем снова на куче из `N` элементов — **1000** `insert` с ключами из отдельного диапазона, чтобы не пересекаться со случайным наполнением. Фиксируются **среднее** время одной операции, **максимум** за одну операцию и **максимум средних** по пакетам из 25 вызовов (сглаживание шума `nanoTime`). Результаты выводятся в `heap_lab_out/` (CSV и PNG).

### Шаг 3. Бинарная min-куча: вставка и просмотр минимума

Новый ключ помещается в конец массива, затем выполняется просеивание вверх; минимум читается из корня.

```24:36:src/main/java/usaskoviy/heaps/BinaryMinHeap.java
    public void insert(int key) {
        ensure(size + 1); // при необходимости расширяем массив под новый элемент
        heap[size] = key; // новый ключ кладём в конец (лист)
        siftUp(size); // проталкиваем вверх, пока не выполнится порядок min-кучи
        size++; // увеличиваем число элементов после фиксации позиции
    }

    public int findMin() {
        if (size == 0) {
            throw new IllegalStateException("empty"); // минимума нет в пустой куче
        }
        return heap[0]; // в min-куче минимум всегда в корне массива
    }
```

### Шаг 4. Биномиальная min-куча: вставка и поиск минимума среди корней

Вставка — дерево из одного узла степени 0, слияние списков корней и устранение совпадающих степеней. Поиск минимума — линейный обход списка корней (их не больше `O(log N)`).

```30:48:src/main/java/usaskoviy/heaps/BinomialMinHeap.java
    public void insert(int key) {
        Node n = new Node(); // новое дерево B_0 — один узел
        n.key = key;
        n.order = 0;
        head = mergeSortedByDegree(head, n); // вливаем в общий отсортированный по степени список корней
        head = unionCarryFromList(head); // устраняем дубликаты степеней (перенос / связывание)
    }

    public int findMin() {
        if (head == null) {
            throw new IllegalStateException("empty"); // в пустой куче минимума нет
        }
        int min = head.key; // кандидат — первый корень в списке
        for (Node cur = head.sibling; cur != null; cur = cur.sibling) {
            if (cur.key < min) {
                min = cur.key; // минимум может быть у любого корня в списке
            }
        }
        return min; // глобальный минимум среди корней (в поддеревьях ключи не меньше по свойству кучи)
    }
```

### Шаг 6. Замер одной серии операций (пример для `findMin`)

Каждая из 1000 операций оборачивается в `nanoTime`; дополнительно считается максимум средних по пакетам из `BATCH` вызовов.

```178:202:src/main/java/usaskoviy/heaps/HeapLabMain.java
    /** Замер 1000 операций findMin: сумма, среднее, макс одной операции, макс среднего по пакетам BATCH. */
    private static Timings measurePeek(BinaryMinHeap h) {
        long total = 0; // суммарное время всех OPS операций (нс)
        long maxSingle = 0; // максимальная длительность одной findMin
        long maxBatchAvg = 0; // максимум средних длительностей по пакетам
        int batches = OPS / BATCH; // число пакетов (1000/25 = 40)
        for (int b = 0; b < batches; b++) {
            long batch = 0; // суммарное время текущего пакета
            for (int i = 0; i < BATCH; i++) {
                long t0 = System.nanoTime(); // начало интервала замера
                h.findMin(); // просмотр минимума без изменения кучи
                long dt = System.nanoTime() - t0; // длительность одной операции
                batch += dt;
                total += dt;
                if (dt > maxSingle) {
                    maxSingle = dt; // обновляем максимум «одного вызова»
                }
            }
            long avgBatch = batch / BATCH; // среднее время внутри пакета (аналог из методички)
            if (avgBatch > maxBatchAvg) {
                maxBatchAvg = avgBatch; // ищем «худший» пакет по среднему
            }
        }
        return new Timings((double) total / OPS, maxSingle, maxBatchAvg); // среднее = total/OPS
    }
```

## 3. Заключение

По **теории** в данной постановке эксперимента бинарная min-куча на массиве выигрывает у биномиальной на операции **`findMin`**: у бинарной минимум в корне массива (**O(1)**), у биномиальной нужен проход по списку корней (**O(log N)** корней). На графиках `heap_avg_findMin.png` и `heap_max_findMin.png` обычно видно, что кривая **бинарной** кучи ниже или стабильнее по времени, особенно при росте `N`.

Для **`deleteMin`** и **`insert`** обе структуры имеют **логарифмическую** асимптотику по `N`, но у бинарной кучи меньше накладных расходов (массив, без объектов на узел), поэтому на `heap_avg_deleteMin.png`, `heap_avg_insert.png` (и соответствующих `heap_max_*.png`) **часто** бинарная куча оказывается быстрее биномиальной при тех же `N`. Точные соотношения нужно смотреть по **вашим** файлам `heap_lab_out/heap_lab_metrics.csv` и PNG после прогона на вашей машине: разброс и «шипы» на кривых максимума связаны с планировщиком ОС, GC и шумом `nanoTime`; колонки `*_batchmaxavg_*` как раз показывают оценку пиков через средние по пакетам.

**Итог работы:** реализованы обе требуемые структуры, соблюдены размеры `N`, по 1000 операций каждого типа, учтены среднее и максимальное время, построены графики зависимости от `N`. Практически бинарная куча на массиве чаще предпочтительна, когда важны простота, память и скорость «точечных» операций с минимумом; биномиальная куча демонстрирует другую модель данных (лес деревьев, слияние корней) и типично сильнее по накладным расходам в Java, что согласуется с курсом «алгоритмы и структуры данных» и с полученными зависимостями на графиках.
